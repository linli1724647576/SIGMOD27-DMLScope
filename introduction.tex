\section{Introduction}
\label{sec:intro}

% 第一段写 DBMS 的背景以及安全漏洞
Database Management Systems (DBMSs) are a fundamental component of modern data-driven applications, ranging from e-commerce and online banking to large-scale enterprise systems~\cite{florescu1998database, zhou2020database}. 
They provide standardized interfaces, such as SQL, to support both data retrieval and data modification~\cite{chamberlin1974sequel}. 
Given their critical role, bugs in DBMSs can lead to severe consequences, including incorrect query results, data corruption, and even security violations~\cite{zhong2020squirrel,jiang2023dynsql}.
Among various types of bugs, logical bugs are particularly critical because they do not crash the system, but silently lead to incorrect query results or incorrect database states, often remaining unnoticed for long periods~\cite{rigger2020detecting, kamm2023testing,song2023testing}. 
Recent studies have demonstrated that even well-tested DBMSs still contain numerous previously unknown logical bugs, highlighting the difficulty of ensuring their correctness~\cite{rigger2020detecting,rigger2020finding,rigger2020testing,song2023testing,tang2023detecting,jiang2024detecting,song2024detecting,hao2023pinolo,deng2025detecting}.

% 第二段写 DML 的实现复杂性， 以及导致出现 Bug，然后说明逻辑错误是尤为严重的 Bug。
While existing research has made significant progress in detecting logical bugs in read-only queries~\cite{rigger2020detecting,rigger2020finding,rigger2020testing,song2023testing,tang2023detecting,jiang2024detecting,song2024detecting,hao2023pinolo,deng2025detecting}, considerably less attention has been paid to state-modifying SQL statements. 
In this work, we focus on data manipulation language (DML) statements that directly change persistent database contents, including common operations such as \texttt{INSERT}, \texttt{UPDATE}, and \texttt{DELETE}, as well as DBMS-specific extensions such as \texttt{REPLACE} when supported. 
Unlike read-only queries that only return result sets, these statements produce persistent state changes, making their correctness harder to validate and their consequences more severe. 
For example, an incorrect \texttt{INSERT} may introduce spurious tuples, an erroneous \texttt{UPDATE} may propagate wrong values to a large number of records, and an unintended \texttt{DELETE} may remove data that should have been preserved, leading to irreversible inconsistencies or data loss. 
Testing such statements is challenging because their semantics are often intertwined with complex predicates, nested subqueries, joins, constraint enforcement, and integrity-related side effects such as cascades. 
As a result, logical bugs in DBMSs may silently affect which tuples are inserted, modified, or deleted, thereby producing incorrect database states without any explicit failure signal~\cite{song2023testing}.

% 第四段写 Detect Logical Bugs 的 Challenges，以及  Existing works
Logical bugs in DBMSs, especially those in DML statements, are difficult to detect automatically.
A fundamental challenge lies in the \emph{test oracle problem}~\cite{howden2006theoretical}, i.e., determining whether the execution of a given SQL statement produces a correct result or database state. 
While this problem has been extensively studied for read-only queries, it becomes significantly more challenging for DML statements, as correctness must be defined over state transitions rather than query outputs. 
To address this challenge, recent work \textsc{DQE}~\cite{song2023testing} proposes a differential testing approach that detects logical bugs by comparing the rows accessed by \texttt{SELECT}, \texttt{UPDATE}, and \texttt{DELETE} statements sharing the same predicate.
The key insight of \textsc{DQE} is that these statements should access the same set of tuples under identical conditions, and any inconsistency indicates a potential logical bug.
Although effective, \textsc{DQE} relies on cross-statement consistency and treats DML correctness indirectly through row-access equivalence.
This design introduces several limitations: (1) it requires restricting SQL features to those commonly supported across different statement types, limiting its applicability to complex queries; (2) it cannot naturally capture value-level correctness (e.g., incorrect assignments in \texttt{UPDATE}); and (3) it may fail to detect bugs when multiple statements share the same erroneous execution path~\cite{hao2023pinolo}.  % 第二、三个局限性怎么说呢?
These limitations suggest that a more direct oracle over database state transitions is needed for DML testing.

% 第五段写其他相关工作
There have also been several studies on detecting logical bugs in DBMSs~\cite{rigger2020detecting,rigger2020finding,rigger2020testing,song2023testing,tang2023detecting,jiang2024detecting,song2024detecting,hao2023pinolo,deng2025detecting}.
For instance, \textsc{TLP}~\cite{rigger2020finding} partitions a query based on three-valued logic and checks equivalence between decomposed queries, while \textsc{NoREC}~\cite{rigger2020detecting} rewrites queries into non-optimized forms to expose inconsistencies.
These techniques are effective at detecting logical bugs in \emph{read-only queries}, where correctness can be defined over result sets.
However, their primary focus is on query equivalence or result-set relations, rather than the correctness of \emph{state-modifying semantics}.
In contrast, our work focuses on logical bugs in DML statements, where errors manifest as incorrect database state transitions, including wrong tuple selection, unintended modifications, and incorrect value assignments.
Such bugs occur during the execution of state-modifying operations and cannot be effectively captured by existing query-centric testing approaches.  % 篇幅不够可以删除

% 第六段写 Key Insight
Inspired by \textsc{Pinolo}~\cite{hao2023pinolo}, which demonstrates that approximate semantic relations can serve as effective invariants for logical bug detection, we revisit this principle in the context of state-changing SQL. 
We term this paradigm \textit{\textbf{A}pproximate \textbf{S}tate-\textbf{R}elation–Driven \textbf{M}etamorphic \textbf{T}esting} (ASRMT): for a statement and its over-/under-approximate variant, correctness is validated by whether their post-execution effects satisfy directional partial-order constraints over database state changes (e.g., subset/superset), rather than output equivalence. 
Any violation of these approximate state-relation invariants indicates a potential logical bug.
