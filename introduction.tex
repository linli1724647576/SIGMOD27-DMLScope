Database Management Systems (DBMSs) are critical infrastructure for modern data-driven applications, ranging from e-commerce and online banking to large-scale enterprise systems~\cite{florescu1998database, zhou2020database}. 
They provide standardized interfaces such as SQL for both data retrieval and data modification~\cite{chamberlin1974sequel}. 
Given their central role, bugs in DBMSs can cause serious consequences, including incorrect query results, data corruption, and even security violations~\cite{zhong2020squirrel,jiang2023dynsql}. 
Among them, logical bugs are particularly dangerous because they do not trigger crashes, but silently produce incorrect results or database states, often remaining hidden for long periods~\cite{rigger2020detecting,kamm2023testing,song2023testing}. 
Recent studies have shown that even mature and widely tested DBMSs still contain many previously unknown logical bugs, highlighting the difficulty of ensuring DBMS correctness~\cite{rigger2020detecting,rigger2020finding,rigger2020testing,song2023testing,tang2023detecting,jiang2024detecting,song2024detecting,hao2023pinolo,deng2025detecting}.

Although prior work has made substantial progress in detecting logical bugs in read-only queries~\cite{rigger2020detecting,rigger2020finding,rigger2020testing,song2023testing,tang2023detecting,jiang2024detecting,song2024detecting,hao2023pinolo,deng2025detecting}, much less attention has been paid to state-modifying SQL statements. 
In this work, we focus on data manipulation language (DML) statements, including \texttt{INSERT}, \texttt{UPDATE}, \texttt{DELETE}, and DBMS-specific extensions such as \texttt{REPLACE}. 
Unlike read-only queries, DML statements directly modify persistent database contents, making their correctness harder to validate and their consequences more severe~\cite{song2023testing}. 
Moreover, DML execution often involves complex predicates, nested subqueries, joins, and integrity-related side effects, which further increase the risk of subtle logical bugs.

Figure~\ref{fig:motivating example} shows a real DML bug from MySQL (Bug \#120047~\cite{mysql_bug_120047}), where the DBMS incorrectly handles row selection during the execution of a join-based \texttt{UPDATE} statement with a complex predicate. 
In this example, the mutated DML joins tables \texttt{t3} and \texttt{t1}, and assigns \texttt{t1.id} to \texttt{t3.ref} for the row in \texttt{t3} that satisfies the \texttt{WHERE} clause. 
However, due to an internal logic error, MySQL fails to update the qualifying row and produces an incorrect final database state. 
This bug does not cause a crash or raise any execution error; instead, the statement appears to execute normally while silently violating its intended semantics.

% 第四段写 Detect Logical Bugs 的 Challenges，以及  Existing works
Logical bugs in DBMSs, especially those in DML statements, are difficult to detect automatically.
A fundamental challenge lies in the \emph{test oracle problem}~\cite{howden2006theoretical}, i.e., determining whether the execution of a given SQL statement produces a correct result or database state. 
While this problem has been extensively studied for read-only queries, it becomes significantly more challenging for DML statements, as correctness must be defined over state transitions rather than query outputs. 
To address this challenge, recent work \textsc{DQE}~\cite{song2023testing} proposes a differential testing approach that detects logical bugs by comparing the rows accessed by \texttt{SELECT}, \texttt{UPDATE}, and \texttt{DELETE} statements sharing the same predicate.
The key insight of \textsc{DQE} is that these statements should access the same set of tuples under identical conditions, and any inconsistency indicates a potential logical bug.
Although effective, \textsc{DQE} relies on cross-statement consistency and treats DML correctness indirectly through row-access equivalence.
This design introduces several limitations: (1) it requires restricting SQL features to those commonly supported across different statement types, limiting its applicability to complex queries; (2) it cannot naturally capture value-level correctness (e.g., incorrect assignments in \texttt{UPDATE}); and (3) it may fail to detect bugs when multiple statements share the same erroneous execution path~\cite{hao2023pinolo}.  % 第二、三个局限性怎么说呢?
These limitations suggest that a more direct oracle over database state transitions is needed for DML testing.

% 第五段写其他相关工作
There have also been several studies on detecting logical bugs in DBMSs~\cite{rigger2020detecting,rigger2020finding,rigger2020testing,tang2023detecting,jiang2024detecting,song2024detecting,hao2023pinolo,deng2025detecting}.
For instance, \textsc{TLP}~\cite{rigger2020finding} partitions a query based on three-valued logic and checks equivalence between decomposed queries, while \textsc{NoREC}~\cite{rigger2020detecting} rewrites queries into non-optimized forms to expose inconsistencies.
These techniques are effective at detecting logical bugs in read-only queries, where correctness can be defined over result sets.
However, their primary focus is on query equivalence or result-set relations, rather than the correctness of state-modifying semantics.
In contrast, our work focuses on logical bugs in DML statements, where errors manifest as incorrect database state transitions, including wrong tuple selection, unintended modifications, and incorrect value assignments.

% 第六段写 Key Insight
Inspired by \textsc{Pinolo}~\cite{hao2023pinolo}, which demonstrates that approximate semantic relations can serve as effective oracles for logical bug detection, we revisit this principle in the context of state-changing SQL. 
Our key insight is that, although the exact post-state of a DML statement is difficult to specify as an oracle, its relative behavior can still be reliably constrained through approximation. 
We therefore term this paradigm \textit{\textbf{A}pproximate \textbf{S}tate-\textbf{R}elation–Driven \textbf{M}etamorphic \textbf{T}esting} (\textbf{ASRMT}). 
Under \model, the semantics of a DML statement are characterized by its state effect, an abstract and comparable representation of the induced state change which we formalize in Section~\ref{sec:asrmt}. 
Given a DML statement and its over- or under-approximate variants, their corresponding state effects are expected to satisfy a approximate relation. 
Any violation of these approximate state-relation invariants indicates a potential logical bug.
% 举个 Motivating Example 的例子，怎么检测 Logical Bug.
For example, in Figure~\ref{fig:motivating example}, we generate a mutated \texttt{UPDATE} statement by transforming the predicate into a logically broader one. 
Under our approximation-based reasoning, the broader statement should induce a state change that subsumes that of the original statement. 
However, MySQL updates one row for the original DML but no row for the mutated one, thereby violating the expected inclusion relation. 
Such a contradiction serves as clear evidence of a logical bug.

% 第七段写方法 DMLScope 的方法
Based on this paradigm, we propose \toolname, a practical testing framework for detecting logical bugs in DML statements. 
\toolname\ instantiates \model\ through a state-oriented generate--mutate--execute--verify workflow. 
Starting from a valid database pre-state, it first generates executable DML statements that satisfy schema and integrity constraints. 
It then synthesizes over- and under-approximate variants using semantics-aware mutators with known approximation directions. 
To ensure meaningful comparison, \toolname\ executes the original and mutated statements independently from the same pre-state under differential state execution. 
After execution, it extracts state effects using representations tailored to the mutation semantics, and finally checks whether the observed effects satisfy the expected approximate state relations. 
Any violation is reported as a potential logical bug.

% 第八段写评估
We evaluate \toolname\ on six widely used and extensively tested DBMSs: MySQL, MariaDB, MonetDB, Percona, DuckDB, and TiDB.
Overall, \toolname\ detected \logical\ logical bugs across all tested systems, including 5 in MySQL, 3 in MariaDB, 9 in MonetDB, 5 in Percona, 1 in DuckDB, and 2 in TiDB. 
In addition, it also triggered 8 abnormal errors and 18 crashes, yielding \reported\ unique bugs in total. 
Among all reported bugs, \confirmed\ have been confirmed by developers, and \fixed\ have already been fixed. 
These results demonstrate the effectiveness of \toolname\ in finding bugs in DBMSs.

Overall, we make the following contributions:
\begin{itemize}[leftmargin=10pt]
	\item We propose Approximate State-Relation--Driven Metamorphic Testing, a novel testing paradigm for detecting logical bugs in state-modifying SQL statements. It characterizes DML semantics through state effects and validates correctness by checking approximate state relations over database state transitions.
	
	\item We implement \toolname, a practical testing framework that systematically generates executable DML statements, synthesizes approximate variants, performs differential state execution, and verifies state-effect relations to detect logical bugs.
	
	\item We evaluate \toolname\ on six real-world DBMSs. In total, we found \confirmed\ bugs including \logical\ logical bugs. To further facilitate research on DBMS testing, we open-source the tool at \url{https://anonymous.4open.science/r/DMLScope-9F03/}.  
\end{itemize}
