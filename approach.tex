\section{Approximate State-Relation–Driven Metamorphic Testing}
\label{sec:asrmt}

We now present \textit{\textbf{A}pproximate \textbf{S}tate-\textbf{R}elation–Driven \textbf{M}etamorphic \textbf{T}esting} (ASRMT), our core paradigm for detecting logical bugs in state-modifying SQL statements. 
Unlike prior approximation-based techniques such as \textsc{Pinolo}~\cite{hao2023pinolo}, which reason about relations over query results, ASRMT defines correctness over database state transitions. 
The key idea is to characterize DML semantics through state effects and to validate correctness by checking approximate relations between the effects of related statements. 
In the following, we formalize state effects, define approximate state relations, and present the corresponding test oracle.

\subsection{State Effect}

Let $S$ denote a database state, and let $q$ be a DML statement. The execution of $q$ over $S$ produces a new state:
\[
\llbracket q \rrbracket(S) \rightarrow S'.
\]

To characterize the observable behavior of a DML statement, we introduce the notion of state effect.

\begin{definition}[State Effect]
	Given a DML statement $q$ and a database state $S$, the state effect of $q$ over $S$, denoted by $E(q,S)$, is an abstract set-valued representation of the observable effect induced by executing $q$ on $S$.
\end{definition}

The notion of state effect is intentionally abstract. In our paper, it is instantiated in two forms, namely the matched-row effect and the changed-row effect, depending on which aspect of the DML semantics is to be observed.

\begin{definition}[Matched-Row Effect]
	Given a DML statement $q$ and a database state $S$, the matched-row effect of $q$, denoted by $E_m(q,S)$, is the set of target-row tokens corresponding to the primary keys of rows structurally matched by $q$ on $S$.
\end{definition}

\begin{example}
	Consider two tables in database state $S$:
	\[
	t_1(c_1, c_2) =
	\begin{array}{c|c}
		c_1 & c_2 \\
		\hline
		1 & 5 \\
		2 & 8 \\
		3 & 9
	\end{array}
	\quad
	t_2(c_1, c_3) =
	\begin{array}{c|c}
		c_1 & c_3 \\
		\hline
		1 & 5 \\
		2 & 20 \\
		3 & 15
	\end{array}
	\]
	
	Consider the DML statement:
	\[
	\begin{aligned}
	q : &\; \text{\textbf{UPDATE} } t_1
	\text{ \textbf{JOIN} } t_2
	\text{ \textbf{ON} } t_1.c_1 = t_2.c_1  
	\text{ \textbf{SET} } t_1.c_2 = t_2.c_3 \\
	&\text{ \textbf{WHERE} } t_2.c_3 > 10
	\end{aligned}
	\]
	
	Let $PK(t_1)=\{c_1\}$ be the primary key of $t_1$.  
	The set of target rows in $t_1$ structurally matched by $q$ is:
	\[
	E_m(q,S)=\{t_1\!:\!2,\; t_1\!:\!3\}.
	\]
	
	Intuitively, the join and predicate select the rows in $t_1$ whose corresponding $t_2.c_3 > 10$, 
	thus determining the intended target set independent of actual value updates.
\end{example}

The matched-row effect captures the \emph{intended target set} of a DML statement. It is suitable when the mutation primarily changes row-selection semantics, such as predicate strengthening/weakening, join-condition mutation, or other transformations that mainly affect which target rows should be selected.

\begin{definition}[Changed-Row Effect]
%	Given a DML statement $q$ and a database state $S$, let $S'=\llbracket q \rrbracket(S)$ be the post-state produced by executing $q$ on $S$. The changed-row effect of $q$, denoted by $E_c(q,S)$, is the set of target-row tokens obtained by comparing $S$ and $S'$ and extracting the rows whose observable states differ before and after execution.
	Given a DML statement $q$ and a database state $S$, the changed-row effect of $q$, denoted by $E_c(q,S)$, is the set of target-row tokens corresponding to the rows whose observable states differ before and after executing $q$ on $S$.
\end{definition}

\begin{example}
	Consider two tables in database state $S$:
	\[
	t1(c_1, c_2) =
	\begin{array}{c|c}
		c_1 & c_2 \\
		\hline
		1 & a \\
		2 & b
	\end{array}
	\quad
	t2(c_1, c_2, c_3) =
	\begin{array}{c|c|c}
		c_1 & c_2 & c_3 \\
		\hline
		5 & x & 1 \\
		6 & y & 2 \\
		7 & z & -1
	\end{array}
	\]
	
	Consider the DML statement:
	\[
	\begin{aligned}
	q : &\; \text{\textbf{INSERT INTO} } t1(c_1, c_2)
	\text{ \textbf{SELECT} } c_1, c_2
	\text{ \textbf{FROM} } t2 \\
	&\text{ \textbf{WHERE} } c_3 > 0
	\end{aligned}
	\]
	
%	Let $S'=\llbracket q \rrbracket(S)$ be the post-state after execution.  
	The newly inserted rows correspond to primary keys $\{5,6\}$.  
	Let $h_5$ and $h_6$ denote their normalized row summaries.  
	We use a normalization function $\mathit{row}(\cdot)$ that maps a tuple to a canonical representation (e.g., a hash of its attribute values).
	Then the changed-row effect is:
	\[
	E_c(q,S)=\{t1\!:\!5\!:\!\mathit{ROW}(h_5),\; t1\!:\!6\!:\!\mathit{ROW}(h_6)\}.
	\]
	
	Intuitively, the effect cannot be derived purely from structure, and must be obtained by comparing the database states before and after execution.
\end{example}
The changed-row effect captures the actual post-execution effect of a DML statement. It is necessary when the mutation may influence value assignments, tuple multiplicity, insertion/deletion behavior, or other effects that cannot be determined reliably from the statement structure alone.

\textbf{Effect Abstraction Selection.}
Accordingly, the abstract state effect $E(q,S)$ used by our approach is instantiated as either $E_m(q,S)$ or $E_c(q,S)$, depending on the comparison strategy. 
The distinction is essential: $E_m(q,S)$ characterizes which target rows are supposed to be affected according to the statement structure, whereas $E_c(q,S)$ characterizes which rows are actually changed in the database after execution.

The choice between $E_m(q,S)$ and $E_c(q,S)$ is not determined merely by the DML type, but by the semantics targeted by the mutation and the robustness of the corresponding effect abstraction. 
In general, $E_c(q,S)$ is more expressive, because it reflects post-execution outcomes and can capture not only row selection but also value updates, insertions, deletions, and multiplicity-related effects. 
However, this stronger expressiveness may also introduce irrelevant execution-level variability. 
For example, in join-based \texttt{UPDATE} statements, one-to-many matches or unstable source-to-target mappings may lead to ambiguous value assignments, even when the intended target-row boundary is clear. 
In such cases, changed-row effects may become noisy or unreliable with respect to the mutation semantics under test. 
By contrast, $E_m(q,S)$ is less expressive but often more stable: it abstracts away concrete value changes and focuses only on the structurally intended target set. 
We next provide a concrete example to illustrate this trade-off and Section~\ref{subsec:State-Effect Extraction} then presents the detailed strategy for selecting the effect abstraction and for extracting state effects in practice.

\begin{example}
	Consider the following tables in database state $S$:
	\[
	t_1(c_1, c_2)=
	\begin{array}{c|c}
		c_1 & c_2 \\
		\hline
		1 & 0 \\
		2 & 0
	\end{array}
	\qquad
	t_2(c_1, c_3)=
	\begin{array}{c|c}
		c_1 & c_3 \\
		\hline
		1 & 100 \\
		1 & 200 \\
		2 & 300
	\end{array}
	\]
	
	Consider the statement
	\[
	\begin{aligned}
		q : &\; \text{\textbf{UPDATE} } t_1
		\text{ \textbf{JOIN} } t_2
		\text{ \textbf{ON} } t_1.c_1 = t_2.c_1 
		\text{ \textbf{SET} } t_1.c_2 = t_2.c_3 
	\end{aligned}
	\]
	
	Structurally, the target rows in $t_1$ matched by $q$ are unambiguous:
	\[
	E_m(q,S)=\{t_1\!:\!1,\; t_1\!:\!2\}.
	\]
	However, the changed-row effect depends on how the one-to-many join for key $c_1=1$ is resolved during execution. 
	If the value $100$ is chosen, one possible changed-row effect is
	\[
	E_c(q,S)=\{t_1\!:\!1\!:\!\mathit{ROW}(h_{100}),\; t_1\!:\!2\!:\!\mathit{ROW}(h_{300})\},
	\]
	whereas if the value $200$ is chosen, another possible changed-row effect is
	\[
	E_c(q,S)=\{t_1\!:\!1\!:\!\mathit{ROW}(h_{200}),\; t_1\!:\!2\!:\!\mathit{ROW}(h_{300})\}.
	\]
	
	This example shows that changed-row effects are more expressive but less stable, since they may incorporate assignment-level ambiguity that is irrelevant when the mutation targets only row-selection semantics. 
	Such ambiguity may arise from DBMS-specific source-row selection policies or execution-dependent factors, such as different join evaluation orders or execution plans, which can lead the same target row to be assigned from different matching source rows.
	In contrast, matched-row effects remain stable because they only record the structurally intended target rows.
\end{example}

\subsection{Approximate State Relation}
To generalize approximation-based metamorphic testing to state-modifying statements, we extend the notion of approximation from query result sets to database state effects.

\begin{definition}[Approximate State Relation]
	\label{def:state_approximation}
	
	Given a database state \( S \), let \( q_1 \) and \( q_2 \) be two DML statements whose state effects are \( E(q_1, S) \) and \( E(q_2, S) \), respectively.  
	We say that \( q_1 \) is the \textit{state-level under-approximation} of \( q_2 \) over \( S \), denoted by
	\[
	q_1 \preceq_S q_2,
	\]
	if and only if
	\[
	E(q_1, S) \subseteq E(q_2, S).
	\]
	Conversely, \( q_1 \) is the \textit{state-level over-approximation} of \( q_2 \) over \( S \), denoted by
	\[
	q_1 \succeq_S q_2,
	\]
	if and only if
	\[
	E(q_1, S) \supseteq E(q_2, S).
	\]
	Here, \( E(q,S) \) denotes the state effect induced by executing statement \( q \) on state \( S \), and \( \subseteq \) and \( \supseteq \) denote inclusion and containment relations between two effect sets.
\end{definition}

\begin{example}
	Consider a table \( t(c_1, c_2) \) in database state \( S \):
	\[
	t =
	\begin{array}{c|c}
		c_1 & c_2 \\
		\hline
		1 & 10 \\
		2 & 20 \\
		3 & 30
	\end{array}
	\]
	Let the following two DML statements be defined:
	\[
	q_1 : \text{\textbf{UPDATE} } t \text{ \textbf{SET} } c_2 = c_2 + 1
	\text{ \textbf{WHERE} } c_1 > 1
	\]
	\[
	q_2 : \text{\textbf{UPDATE} } t \text{ \textbf{SET} } c_2 = c_2 + 1
	\text{ \textbf{WHERE} } c_1 \ge 1
	\]
	If the state effect is instantiated as the matched-row effect, then
	\[
	E(q_1, S) = \{t\!:\!2,\; t\!:\!3\}, \qquad
	E(q_2, S) = \{t\!:\!1,\; t\!:\!2,\; t\!:\!3\}.
	\]
	Since
	\[
	E(q_1, S) \subseteq E(q_2, S),
	\]
	we have
	\[
	q_1 \preceq_S q_2.
	\]
	Intuitively, \( q_2 \) weakens the selection predicate and therefore affects a superset of the rows affected by \( q_1 \).
\end{example}

This definition lifts set-semantic approximation from query outputs to state transitions, enabling reasoning about correctness in terms of persistent database effects.
To systematically construct DML statements with predictable approximation relations, we introduce approximate mutators.

\begin{definition}[Approximate Mutator]
	\label{def:approx_mutator}
	
	Let $q$ be a DML statement and let $n$ be a syntactic construct in $q$.  
	An \textit{approximate mutator} $\mu$ transforms $q$ into a new statement $q' = \mu(q)$ by modifying $n$ such that:
	\[
	q \preceq_S q' \quad \text{or} \quad q \succeq_S q'.
	\]
	The direction of the induced relation determines whether $\mu$ is an under- or over-approximate mutator.
\end{definition}

\begin{example}
	Consider the statement
	\[
	q : \text{\textbf{DELETE FROM} } t
	\text{ \textbf{WHERE} } c_1 > 1.
	\]
	
	Let \(n\) be the comparison predicate \(c_1 > 1\), and let \(\mu\) be an approximate mutator that weakens the comparison operator \(>\) to \(\ge\). 
	Applying \(\mu\) to \(q\) yields
	\[
	q' = \mu(q) : \text{\textbf{DELETE FROM} } t
	\text{ \textbf{WHERE} } c_1 \ge 1.
	\]
	
	Since this transformation enlarges the set of affected rows, we have
	\[
	q \preceq_S q'.
	\]
	Therefore, \(\mu\) is an over-approximate mutator in this example.
\end{example}

Approximate mutators provide a way to generate related statements whose semantic relationships are known \emph{a priori}, which is essential for constructing effective metamorphic tests.
However, a mutation typically affects only a local component of a statement. We therefore introduce approximation propagation to characterize how local changes determine global state-effect relations.

\begin{definition}[Approximation Propagation]
	\label{def:approx_propagation}
	
	Let $q$ be a DML statement and let $q'$ be obtained by applying an approximate mutator to a local construct $n \in q$.  
	Approximation propagation describes how the local semantic deviation induced at $n$ determines a global approximate state relation between $q$ and $q'$:
	\[
	q \preceq_S q' \quad \text{or} \quad q \succeq_S q'.
	\]
\end{definition}

\begin{example}
	Consider a table $t(c_1, c_2)$ in database state $S$:
	\[
	t =
	\begin{array}{c|c}
		c_1 & c_2 \\
		\hline
		1 & 5 \\
		2 & 10 \\
		3 & 15
	\end{array}
	\]
	
	Consider the DML statement
	\[
	q : \text{\textbf{DELETE FROM} } t
	\text{ \textbf{WHERE} } \neg (c_2 > 10).
	\]
	
	If the state effect is instantiated as the matched-row effect, then the affected rows are
	\[
	E(q,S)=\{t\!:\!1,\; t\!:\!2\}.
	\]
	since the predicate $\neg(c_2 > 10)$ is equivalent to $c_2 \le 10$.
	
	Now consider a local mutation that replaces the comparison operator $>$ with $\ge$ inside the negation:
	\[
	q' : \text{\textbf{DELETE FROM} } t
	\text{ \textbf{WHERE} } \neg (c_2 \ge 10).
	\]
	
	Although the local mutation $>$ $\rightarrow$ $\ge$ weakens the comparison expression itself, 
	the surrounding negation reverses the approximation direction. 
	Indeed, $\neg(c_2 \ge 10)$ is equivalent to $c_2 < 10$, and thus
	\[
	E(q',S)=\{t\!:\!1\}.
	\]
	
	Therefore,
	\[
	E(q',S) \subseteq E(q,S),
	\]
	which means that $q'$ is a state-level under-approximation of $q$. 
	This example illustrates that a local mutation may induce the opposite approximation direction at the statement level when it is propagated through a negation context.
\end{example}

Approximation propagation connects local syntactic transformations with global semantic relations, ensuring that approximation reasoning remains sound at the level of full DML statements.

\subsection{Test Oracle}

Based on the above definitions, \model\ uses inclusion relations over state effects as the test oracle.

%\begin{definition}[State-Relation Oracle]
%	Given a database state $S$, a seed DML statement $q$, and its approximate variant $q'$,  
%	we say that a logical bug is detected if the observed state effects violate their expected approximate state relation.  
%	More specifically:
%	
%	\[
%		\text{if } q \preceq_S q', \text{ then a bug is detected whenever } E(q,S) \nsubseteq E(q',S);
%	\]
%	
%	\[
%		\text{if } q \succeq_S q', \text{ then a bug is detected whenever } E(q,S) \nsupseteq E(q',S).
%	\]
%\end{definition}
\begin{definition}[State-Relation Oracle] 
	Given a seed statement $q$ and its approximate variant $q'$, a logical bug is detected if the observed state effects violate the expected inclusion relation: 
	\begin{itemize} 
		\item Over-approximation: $E(q, S) \nsubseteq E(q', S)$ 
		\item Under-approximation: $E(q', S) \nsubseteq E(q, S)$ 
	\end{itemize} 
\end{definition}

This formulation enables approximation-based reasoning over persistent database behaviors, extending metamorphic testing from query correctness to state-transition correctness.

\textbf{Discussion.}
\model\ differs fundamentally from prior approximate metamorphic testing framework \textsc{Pinolo}~\cite{hao2023pinolo}, which target read-only queries and define approximation relations over result sets. 
For DML statements, correctness is determined not by query outputs, but by how execution changes the underlying database state. 
This shift requires redefining the semantic object of approximation as state effect and reasoning about approximation in terms of state transitions. 
Moreover, because DML semantics involve both intended target-row selection and actual post-execution changes, \model\ must support multiple effect abstractions and propagate local mutations under state-based semantics.


\section{Approach}
\label{sec:approach}

Based on the \model\ model, we design \toolname, a practical testing framework for detecting logical bugs in DML statements.
The key idea of \toolname\ is to instantiate approximate state relations over executable DML statements, execute the original and mutated statements from the same initial database state, and check whether their observed state effects satisfy the expected inclusion relation.
In the following, we present the overall workflow of \toolname\ and then describe each core component in detail.

\subsection{Overview}

Figure~\ref{fig:overview} shows the overall workflow of \toolname.
The entire process follows a state-oriented generate--mutate--execute--verify paradigm.
Starting from a round-specific database pre-state, \toolname\ first generates an executable seed DML statement, then synthesizes its approximate variants according to the approximation relations defined in Section~\ref{sec:asrmt}.
Next, all statements are executed under differential state execution, where each statement is executed independently from the same pre-state and the database state is restored after each execution, so that their resulting state effects are directly comparable.
Finally, \toolname\ extracts state-effect tokens and checks whether the observed effects satisfy the expected approximate state relation.
Any violation of the expected relation indicates a potential logical bug.

\subsection{Database Initialization}
\label{subsec:Database Initialization}
Before generating and testing DML statements, \model\ first constructs an independent round environment by initializing a database with a valid schema and initial data. 
This step is not a primary contribution of our work, but is necessary to ensure the correctness and executability of subsequent testing stages.
Specifically, we first generate a relational schema consisting of multiple tables with primary key and foreign key constraints, and map logical column types to DBMS-specific physical types to ensure compatibility across different systems. 
We then populate each table with a small number of rows while satisfying integrity constraints such as key uniqueness and referential dependencies. 
The insertion order follows dependency relationships so that referenced tables are initialized before dependent ones. 
In addition, we maintain auxiliary metadata, such as the set of existing primary keys, to guide the generation of valid DML statements and avoid referencing non-existent tuples. 
This design ensures that all generated statements are executable on the constructed database state and provides a stable pre-state for subsequent differential state execution.

\subsection{Original DML Generation}
\label{subsec:Original DML Generation}

To systematically explore diverse state-modifying behaviors, we first generate syntactically valid and executable DML statements as testing seeds.
This process consists of two stages: \emph{Grammar-guided structure generation}, which constructs abstract syntax trees (ASTs) according to a predefined grammar, and \emph{Schema-based instantiation}, which binds schema elements and concrete values to produce executable statements.
Compared with read-only query generation, original DML generation must additionally guarantee the validity of state-modifying semantics. 
If a DML statement that violates integrity constraints, such as \texttt{PRIMARY KEY}, \texttt{UNIQUE}, \texttt{NOT NULL}, or \texttt{FOREIGN KEY}, may abort entirely or cause rollback, making its state effect undefined and therefore unusable for subsequent approximation checking. 
Algorithm~\ref{alg:original_dml_generation} summarizes the overall procedure. 

\textbf{Grammar-Guided Structure Generation.}

%The grammar covers the major DML categories, including \texttt{UPDATE}, \texttt{DELETE}, \texttt{INSERT}, and \texttt{REPLACE}, as well as rich query constructs such as joins, nested subqueries, common table expressions (CTEs), and set operations.

Initially, the algorithm performs \emph{grammar-guided structure generation} (Lines~1--4), which determines the abstract syntactic shape of a seed DML statement.
It first selects a statement kind $k$ (Line~2), then chooses a compatible target table $T$ (Line~3), and finally expands the corresponding grammar rules to build an abstract statement skeleton $A$ (Line~4).
We use the compact DML grammar in Figure~\ref{fig:dml-bnf-compact} to guide this process.
Beyond basic DML forms, the grammar also supports joins, nested subqueries, common table expressions, and set operations, enabling the generation of structurally rich statements.
The generator recursively samples productions to construct an AST, while bounding recursion depth and restricting highly combinatorial constructs to keep generation tractable.
At this stage, only the structural skeleton is determined; concrete schema bindings and values are instantiated later.

\textbf{Schema-based Instantiation.}
Next, the algorithm performs \emph{schema-based instantiation}, which concretizes the abstract skeleton into an executable DML statement under the current schema and database pre-state.
It begins by generating candidate target columns from the skeleton and the target-table schema (Line~6), and then filters out invalid columns that are not writable, not visible in the current scope, or incompatible with the chosen statement kind (Line~7).
Based on the remaining column candidates, the algorithm generates source candidates that may provide values, bindings, or filtering contexts for the statement (Line~8), and removes unstable or incompatible sources by enforcing source-side semantic constraints (Line~9).
It then materializes a consistent column--source plan $(\pi_c,\pi_s)$ (Line~10), which serves as the blueprint for the subsequent instantiation process.
After the plan is fixed, the algorithm binds concrete schema elements to the skeleton (Line~11), and then incrementally instantiates key positions, non-key values, and predicates (Lines~12--14).
This order is important.
Key positions are instantiated first because they determine tuple identity and may immediately conflict with the current pre-state.
Once key values are fixed, the algorithm instantiates non-key payloads under source, type, and nullability constraints, and finally fills predicate slots with scope-correct and semantically meaningful filtering conditions.
In this way, the generator does not first produce an arbitrary statement and then attempt to repair it afterwards; instead, it progressively constructs the statement while preserving semantic consistency at each stage.
The algorithm then performs DML-specific semantic checks according to the statement kind (Lines~15--22).
For \texttt{INSERT} and \texttt{REPLACE}, it checks key consistency and source--payload alignment (Lines~15--17), ensuring that inserted tuples neither violate uniqueness constraints nor introduce incompatible source-to-target mappings.
For \texttt{UPDATE}, it checks assignment validity (Lines~18--19), so that writable target columns, source bindings, and right-hand-side expressions remain jointly meaningful.
For \texttt{DELETE}, it checks that the predicate is non-degenerate (Lines~20--21), avoiding statements that collapse into trivial or uncontrolled deletions.
These statement-specific checks are essential because, unlike read-only query generation, DML generation must additionally preserve the validity of state-modifying semantics.
Finally, the algorithm performs structural checking, serialization, and execution validation (Lines~23--28).
It first verifies that the instantiated AST remains structurally complete and internally consistent (Line~23), and then serializes it into a concrete SQL statement $q$ (Line~24).
The statement is accepted only if it can be executed successfully on the current pre-state $S$ (Lines~25--26); otherwise, the generation process restarts (Lines~27--28).

\subsection{Approximate DML Synthesis}
\label{subsec:Approximate DML Synthesis}


Given an executable original DML statement, \toolname\ next synthesizes a set of approximate variants whose state effects are expected to follow a known approximate relation with respect to the seed statement.
Instead of generating new statements from scratch, the algorithm systematically applies approximate mutators to selected components of the original DML.

Table~\ref{tab:dml_mutators} summarizes some representative approximate mutators used in our synthesis phase.
Our design follows the general spirit of \textsc{Pinolo}~\cite{hao2023pinolo}, which organizes mutators around semantic approximation, while extending it to better support DML-specific constructs.
In particular, beyond relation-, predicate-, and comparison-oriented mutations, we additionally introduce \emph{Join / Structure} mutators to handle DML-relevant constructs such as join type and \texttt{HAVING}-related query structure.
Overall, compared with \textsc{Pinolo}, our framework introduces \textcolor{red}{XX} additional mutators; more complete mutation rules are available in our GitHub repository.

Algorithm~\ref{alg:approx_dml_synthesis} summarizes the overall approximate DML synthesis procedure.
Given an executable original DML statement \(q\), the algorithm first parses it into an AST and identifies a set of semantically meaningful local constructs that admit approximate mutation (Lines~2--3), including predicates, comparison expressions, join conditions, and query-structure constructs.
It then iterates over these constructs and enumerates applicable approximate mutators for each of them (Lines~6--8).
For every selected mutator, the algorithm applies the mutation at the AST level and serializes the resulting structure into a concrete DML statement \(q'\) (Lines~9--10).
Next, it determines the approximation direction in two stages: it first infers the local direction induced at the mutation site by the selected mutator (Line~11), and then propagates this direction through the surrounding syntactic context to obtain the global approximation direction of the mutated statement (Line~12).
In particular, when the mutation site appears in direction-reversing contexts, such as \texttt{NOT}, \texttt{NOT EXISTS}, or the right branch of \texttt{EXCEPT}, the approximation direction is reversed, following the same propagation principle as in \textsc{Pinolo}~\cite{hao2023pinolo}.

\begin{algorithm}[t]
	\caption{Approximate DML Synthesis}
	\footnotesize
	\label{alg:approx_dml_synthesis}
	\begin{algorithmic}[1]
		\Require Original DML statement $q$
		\Ensure A set of approximate variants $\mathcal{R}$
		
		\State \textbf{// Step 1: Parse and collect mutation candidates}
		\State Parse $q$ into AST $A$
		\State $\Gamma \gets \textsc{CollectMutableNodes}(A)$
		
		\State \textbf{// Step 2: Apply approximate mutators}
		\State $\mathcal{R} \gets \emptyset$
		\ForAll{node $n \in \Gamma$}
		\State $\Omega \gets \textsc{ApplicableMutators}(n)$
		
		\ForAll{mutator $\mu \in \Omega$}
		\State $A' \gets \textsc{ApplyMutation}(A, n, \mu)$
		\State $q' \gets \textsc{Serialize}(A')$
		
		\State $m_\ell \gets \textsc{InferLocalDirection}(\mu)$
		\State $m \gets \textsc{PropagateDirection}(m_\ell, n, A)$
		\State $\mathcal{R} \gets \mathcal{R} \cup \{(q', m)\}$
		\EndFor
		\EndFor
		
		\State \Return $\mathcal{R}$
	\end{algorithmic}
\end{algorithm}

\subsection{Differential State Execution}
\label{subsec:Differential State Execution}

To make state effects directly comparable across related statements, \toolname\ executes the original DML statement and all its approximate variants under differential state execution. 
The key requirement is that every statement must be evaluated from the same database pre-state \(S\), rather than from a state already modified by another execution.
This requirement is essential for \model\, because the expected approximate state relation is defined between \(E(q,S)\) and \(E(q',S)\), i.e., the state effects of two statements over the same initial state. 
If one statement is executed after another without restoring the database, the resulting effect would instead be measured over a different state, making the subsequent comparison semantically invalid.

Algorithm~\ref{alg:differential_state_execution} summarizes the overall differential state execution procedure.
Given a database pre-state \(S\), an original DML statement \(q\), and its synthesized approximate variants \(\mathcal{R}\), the algorithm first materializes a statement set \(\mathcal{Q}\) consisting of the original statement and all mutated statements (Line~1).
It then iterates over each statement \(q_i \in \mathcal{Q}\) and executes it independently from the same pre-state (Lines~3--6).
Concretely, before each execution, the algorithm restores the database to \(S\) (Line~4), so that every statement observes an identical input state.
It then executes the current statement to obtain its post-state \(S_i'\) (Line~5) and records this post-state as the independent execution result of \(q_i\) (Line~6).
This design isolates each execution from the side effects of all others, ensuring that the post-states later used for state-effect extraction are produced from a common starting point rather than from states modified by previous executions.
In practice, we implement the restoration step through dual isolation. 
First, each statement is executed in a separate subprocess, which isolates executions at the runtime level and prevents side effects from propagating through shared process state. 
Second, within each subprocess, the statement is executed inside an isolated transaction and the transaction is rolled back after execution, thereby restoring the database to the round-level pre-state \(S\). 
Together, these two mechanisms ensure that every original or approximate DML statement is evaluated from the same clean execution environment.
The resulting post-states are then passed to the next stage for state-effect extraction.

\begin{algorithm}[t]
	\caption{Differential State Execution}
	\footnotesize
	\label{alg:differential_state_execution}
	\begin{algorithmic}[1]
		\Require Database pre-state $S$, original DML statement $q$, approximate variants $\mathcal{R}$
		\Ensure A set of post-states $\mathcal{X} = \{(q_i, S_i')\}$
		
		\State $\mathcal{Q} \gets \{q\} \cup \{q' \mid (q', m) \in \mathcal{R}\}$
		\State $\mathcal{X} \gets \emptyset$
		\ForAll{statement $q_i \in \mathcal{Q}$}
		\State \textsc{RestoreState}$(S)$
		\State $S_i' \gets \llbracket q_i \rrbracket(S)$
		\State $\mathcal{X} \gets \mathcal{X} \cup \{(q_i, S_i')\}$
		\EndFor
		\State \Return $\mathcal{X}$
	\end{algorithmic}
\end{algorithm}

\subsection{State-Effect Extraction}
\label{subsec:State-Effect Extraction}

After differential state execution, \toolname\ extracts state-effect tokens from each executed statement, which serve as the canonical representation for subsequent relation checking.
Algorithm~\ref{alg:state_effect_extraction} summarizes the overall state-effect extraction procedure.

Given the execution results $\mathcal{X} = \{(q_i, S_i')\}$ produced in Section~\ref{subsec:Differential State Execution}, the algorithm processes each statement independently.
For each $q_i$, it first parses the statement into a structured representation (Line~2), which includes its statement kind, target table, primary key, and (if available) a derived matching query (\texttt{match\_sql}).
Based on the mutation operator and its syntactic context, the algorithm then determines an extraction strategy (Lines~3--5), which specifies both the extraction basis (\texttt{matched\_rows} or \texttt{changed\_rows}) and whether only primary keys should be considered.
% 怎么判断选择 changed rows 或者 matched rows的呢
Specifically, \toolname\ adopts a stability-aware selection strategy for effect extraction.
It first executes the derived \texttt{match\_sql} query and analyzes the mapping cardinality between source tuples and target primary keys.
If multiple matched rows correspond to the same target primary key, the statement exhibits a one-to-many mapping from source-side matches to target rows.
In such cases, the final value written to a target row may depend on DBMS-specific execution details, such as join order or plan selection, making the concrete state change unstable or non-deterministic.
To avoid introducing such execution-level variability into the oracle, the algorithm selects the \texttt{matched\_rows} strategy, which focuses only on the structurally matched target-row set.
Otherwise, when each target row corresponds to at most one matched source tuple, the induced value assignment is deterministic, and the algorithm selects the \texttt{changed\_rows} strategy to capture precise value-level state changes.
%  提取策略
The extraction follows two alternative paths depending on the selected strategy.
If the \texttt{matched\_rows} strategy is applicable, the algorithm derives the target-row set directly from the statement structure (Lines~6--7).
Specifically, it executes the derived \texttt{match\_sql} query to obtain the primary keys of all structurally matched rows, and converts them into tokens of the form $\texttt{table:pk}$.
This path avoids dependence on execution results and is therefore more stable when the mutation primarily affects row-selection semantics.
Otherwise, the algorithm adopts the \texttt{changed\_rows} strategy (Lines~8--11), which extracts effects from actual state differences.
It first reconstructs the pre-state snapshot $S$ and compares it with the post-state $S_i'$ obtained from differential execution.
Rows whose observable representations differ between the two states are identified as changed rows.
Each such row is normalized into a token of the form $\texttt{table:pk:ROW(hash)}$, capturing both row identity and value changes.
This strategy is necessary when the mutation may influence value assignments, tuple multiplicity, or insertion/deletion behavior.
%Finally, all extracted tokens are normalized into a set representation (Line~12), referred to as the state-effect token set.
Finally, all extracted tokens are normalized into a set representation (Line~12), referred to as the state-effect token set, which serves as a concrete representation of the state effect \(E(q_i,S)\).
These token sets abstract away DBMS-specific execution details while preserving the essential semantics required for approximation checking, and are used in the next stage to verify expected over- or under-approximation relations.

% 给两个例子 Why Select matched row, how to extract effect

\begin{algorithm}[t]
	\caption{State-Effect Extraction}
	\footnotesize
	\label{alg:state_effect_extraction}
	\begin{algorithmic}[1]
		\Require Execution results $\mathcal{X} = \{(q_i, S_i')\}$, pre-state $S$, mutation context $(\mu, n)$
		\Ensure State-effect token sets $\{\mathcal{T}_i\}$
		
		\ForAll{$(q_i, S_i') \in \mathcal{X}$}
		\State $dml \gets \textsc{ParseDml}(q_i)$
		\State $basis \gets \textsc{SelectBasis}(\mu, n, dml, S)$
		\State $pk\_only \gets \textsc{NeedPKOnly}(\mu, n, basis)$
		\State $strategy \gets (basis, pk\_only)$
		
		\If{$basis = \texttt{matched\_rows}$}
		\State $rows \gets \textsc{Execute}(dml.match\_sql)$
		\State $\mathcal{T}_i \gets \textsc{MatchToTokens}(rows, pk\_only)$
		\Else
		\State $before \gets \textsc{Snapshot}(S)$
		\State $after \gets \textsc{Snapshot}(S_i')$
		\State $\mathcal{T}_i \gets \textsc{DiffToTokens}(before, after, pk\_only)$
		\EndIf
		\EndFor
		
		\State \Return $\{\mathcal{T}_i\}$
	\end{algorithmic}
\end{algorithm}

Figure~\ref{fig:state_effect_example} illustrates two representative scenarios for state-effect extraction.
Consider the \texttt{UPDATE} statement shown in Figure~\ref{fig:state_effect_example}(a), where the mutation only affects the join predicate (e.g., $t1.c1 \le t2.c1$).
Although different executions may produce different assigned values due to multiple matching rows in $t2$, the set of affected target rows in $t1$ remains invariant.
In this case, extracting state effects from execution results would introduce unnecessary instability.
Therefore, the algorithm selects the \texttt{matched\_rows} strategy and derives a matching query (\texttt{match\_sql}) from the statement structure.
By executing this query, it obtains the set of structurally matched primary keys $\{1,2\}$, which directly forms the state effect $E(q,S)$.
In contrast, Figure~\ref{fig:state_effect_example}(b) shows an \texttt{INSERT} statement, where the mutation influences both tuple generation and value assignments.
Here, the affected rows cannot be determined purely from the query structure, as the inserted tuples depend on the execution outcome.
Thus, the algorithm selects the \texttt{changed\_rows} strategy.
It compares the pre-state $S$ and the post-state $S1$ to identify newly inserted rows (e.g., $(4,40)$), and normalizes them into tokens of the form $\texttt{table:pk:ROW(hash)}$.
These tokens capture both row identity and value changes, yielding the state effect $E(q,S)$.

\subsection{Approximate State Relation Checking}
\label{subsec:Approximate State Relation Checking}

After state-effect extraction, \toolname\ verifies whether the extracted token sets satisfy the expected approximate state relation.
For an over-approximate mutant, the token set of the original statement should be contained in that of the mutant, whereas for an under-approximate mutant, the token set of the mutant should be contained in that of the original statement.
Any violation of the expected relation indicates a potential logical bug.
